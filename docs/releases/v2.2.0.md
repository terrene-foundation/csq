# csq v2.2.0

**Release date:** 2026-04-25
**Prior release:** v2.1.1 (2026-04-24)
**Checksum / signature:** see artifact `SHA256SUMS` and `.sig` files on the GitHub Releases page.

## What's new

v2.2.0 is a **minor release** on v2.1.1. Two backwards-compatible onboarding improvements reported against fresh WSL installs the day after the v2.1.1 cut:

- **`csq status` now shows every configured surface**, not just Anthropic OAuth slots. Codex, per-slot third-party (MiniMax / Z.AI / Ollama), global third-party bindings, and manually configured rows are all visible, each rendered with a surface tag.
- **`csq install` and `csq run` pre-emptively detect** the two failure modes behind the `SessionStart:startup hook error` / `node:internal/modules/cjs/loader:1143` noise new users keep hitting on fresh WSL.

No schema changes, no credential migration, no breaking changes to CLI flags, command output parsing, or IPC contracts. `AccountStatus` JSON adds two optional fields; older csq snapshots deserialise via serde defaults.

## The changes

### Multi-surface `csq status` (#189-follow-up)

**Before:** `csq status` used `discovery::discover_anthropic` directly. Users with Codex OAuth slots (landed in v2.1.0), per-slot 3P bindings (MiniMax / Z.AI / Ollama), or manually configured rows saw those slots silently dropped from the listing. The daemon's `GET /api/accounts` had the same gap — the cached discovery path also walked Anthropic-only.

**After:** `csq status` and the daemon route both use `discovery::discover_all`, composing Anthropic OAuth + Codex + per-slot 3P + global 3P + manual rows in priority order (with deduplication by slot ID). The `csq-core::quota::status::AccountStatus` struct gains two fields:

- `source: AccountSource` — Anthropic OAuth, Codex OAuth, `ThirdParty { provider }`, or Manual.
- `surface: Surface` — `claude-code` or `codex`.

Both fields carry `#[serde(default)]` so pre-2.2.0 JSON snapshots parse cleanly. `AccountStatus::format_line` renders a surface tag after the label:

```text
* #1 ● alice@example.com              5h:25% (45m) 7d:8%
  #4 ● openai@example.com [codex]     5h:12% (30m) 7d:3%
  #9 · MiniMax [minimax]              (api-key)
  #10 · Ollama [ollama]               (api-key)
```

Anthropic OAuth rows render byte-identical to pre-2.2.0 output (empty surface tag, unchanged quota suffix), so any downstream scripts or dashboards that parsed `csq status` stdout keep working without adjustment. Third-party / manual rows skip the `5h:— / 7d:—` placeholder and render `(api-key)` instead — csq does not poll those providers' quotas today, and "no polling" needs to be distinguishable from "no data yet".

### Hook-error preflight (issue-free, reported out-of-band)

New users on fresh WSL Ubuntu images keep reporting:

```text
SessionStart:startup hook error Failed with non-blocking status code
node:internal/modules/cjs/loader:1143
```

The error is Claude Code reporting a hook failure; it's attributed to csq because csq is the visible entry point (users see it after `csq install` / `csq run N`). csq itself never writes hook configuration into `~/.claude/settings.json`, but it should still surface the actionable remediation before the error lands mid-session.

**Two root causes, now both detected:**

1. **Node.js not installed.** WSL Ubuntu base images ship without `node`; the first hook invocation emits `node: command not found`.
2. **Hook `require("./lib/...")` fails.** A top-level hook file was materialised without its sibling helper modules (common when a project template was partially synced, or when the hook script was hand-copied out of `scripts/hooks/` without `lib/`). This is the precise `loader:1143` stack line users report.

**The fix** is a new `csq_core::platform::env_check` module that runs a pre-flight against both the global `~/.claude/settings.json` and the project `$CWD/.claude/settings.json`:

- Parses every configured hook's command line, expands `$CLAUDE_PROJECT_DIR` and `~`, and verifies the script exists.
- For `.js` scripts, scans the file for `require("./relative")` patterns and resolves each target with the same extension probing node does (`.js`, `.cjs`, `.mjs`, `.json`, `index.js`). Unresolved targets surface as `HookRelativeRequireMissing`.
- Detects the current platform: macOS (brew) / Windows (winget) / Debian-Ubuntu / WSL / RHEL-Fedora / Arch. Returns the right one-line install command per flavor.

`csq install` runs the preflight after settings patching, reports every issue, and on Linux-family flavors prompts `[y/N]` before running `sudo apt install -y nodejs` (or `dnf` / `pacman`). macOS / Windows / unknown Linux flavors print the hint command without prompting — csq never runs `sudo` silently.

`csq run` runs the same preflight at the top of `handle()`, stderr-only and non-blocking — users have already decided to launch, so mid-session interactive prompts would strand them. The warning text still names the exact fix.

README Troubleshooting gains a dedicated entry for the error string with per-platform commands.

## Migration & compatibility

**No schema migration.** `quota.json` stays at v2. Credential files, handle dirs, `settings.json` files, and daemon socket protocol are compatible both directions between v2.1.1 and v2.2.0. The added `source` / `surface` fields on `AccountStatus` deserialise via serde default on old snapshots.

**No behaviour change for Anthropic-only users.** `csq status` on a machine with only Anthropic OAuth slots produces byte-identical output to v2.1.1. The new surface tags (`[codex]`, `[minimax]`, `[zai]`, `[ollama]`, `[manual]`) only render for rows whose source is non-Anthropic.

**Preflight is additive.** On environments where node is present and hooks resolve correctly, the preflight is silent — neither `csq install` nor `csq run` changes its existing output.

**No auto-install without consent.** Even on Linux flavors where csq knows the package-manager command, the preflight prompts `[y/N]` before running `sudo`. A script piping `echo "" | csq install` (EOF without input) defaults to "no" — csq never escalates privileges silently.

## Platform notes

Same as v2.1.1 — no Windows-side code changes in this minor. Windows daemon (L4) and macOS ad-hoc signature (L1) limitations carry over unchanged. Codex on Windows remains unsupported pending PR-VP-C1b (the `windows-daemon` Cargo feature flip). See `docs/releases/v2.1.0.md` §Platform notes for the full Windows status.

The preflight's WSL detection uses `/proc/version` containing "microsoft" or "WSL" — this is the documented WSL detection pattern and survives both WSL1 and WSL2 kernels.

---

## Limitations that carry over

L1 (macOS ad-hoc signature), L2 (`csq update install` UX coverage), L3 (Windows SmartScreen), L4 (Windows daemon not wired by default). See `docs/releases/v2.0.0.md` for detail.

## Upgrading

```
# Existing v2.1.x users (tray)
Click "Update" in the csq tray menu.

# CLI users
curl -L https://github.com/terrene-foundation/csq/releases/download/v2.2.0/csq-<os>-<arch> -o /usr/local/bin/csq
chmod +x /usr/local/bin/csq
```

After upgrade:

- `csq status` will start showing your Codex / MiniMax / Z.AI / Ollama / manual slots alongside Anthropic OAuth slots. No user action needed.
- `csq install` and `csq run` will now report any hook-environment issues they detect. If you had no issues before, nothing changes. If you had a silent `loader:1143` error on every CC launch, you'll now see the exact fix command on the next csq install / csq run.
