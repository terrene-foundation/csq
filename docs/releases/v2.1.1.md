## What's new

v2.1.1 is a **patch release** on v2.1.0. It closes two on-disk-artifact migration gaps reported the day after the v2.1.0 cut. No new features; no schema changes; no behavior change for fresh installs.

If you upgraded to v2.1.0 from any earlier csq, update.

### The fixes

#### Issue #184 — strip legacy `apiKeyHelper` from 3P settings

Pre-alpha.8 csq serialized `Provider::system_primer` (a long English instruction string) under a top-level `apiKeyHelper` key in third-party provider settings (`config-<N>/settings.json`, `settings-mm.json`, `settings-zai.json`). Claude Code interprets `apiKeyHelper` as a SHELL COMMAND that prints an API key, so every CC launch on an affected slot emitted:

```text
apiKeyHelper failed: exited 127: /bin/sh: You: command not found
⚠ Auth conflict: Both a token (ANTHROPIC_AUTH_TOKEN) and an API key (apiKeyHelper) are set.
```

The write paths were hardened in alpha.8 (regression test `bind_strips_api_key_helper`; `providers/settings.rs` no longer emits the field), but **on-disk artifacts on upgraded machines were never cleaned up**. The user reporting the issue had three contaminated files on a single machine — the production landmine survived across every csq upgrade since alpha.7.

**Fix:** new `pass4` in the daemon's startup reconciler (`csq-core/src/daemon/migrate_legacy_api_key_helper.rs`). Walks `<base_dir>/config-<N>/settings.json` and `<base_dir>/settings-*.json`, strips `apiKeyHelper` only when both `apiKeyHelper` AND `env.ANTHROPIC_AUTH_TOKEN` are present (the unambiguous legacy-bug signature; csq itself never wrote any other shape and a hypothetical user-authored helper script alone is preserved). Atomic write via `unique_tmp_path` + `secure_file` (clamps to 0o600) + `atomic_replace`; per `security.md` §5a every failure branch removes the umask-default tmp file before propagating the error so a partial migration never leaves a token-bearing file at world-readable perms. Idempotent. Preserves mtime on no-op so CC doesn't re-stat for nothing (spec 01 §1.4).

The migration runs the next time the daemon starts (typically within seconds of opening the desktop app or running `csq run N`). No user action required; structured-logged at INFO with `error_kind = "migrate_strip_api_key_helper"` per file rewrite for operator traceability.

#### Issue #185 — extend `csq install` statusline migration to handle dirs

`csq install` migrated the legacy `bash ~/.claude/accounts/statusline-quota.sh` wrapper out of global `~/.claude/settings.json` and per-slot `config-<N>/settings.json`, but did NOT walk per-terminal handle dirs at `~/.claude/accounts/term-*/settings.json`. `cleanup_v1_artifacts` then renamed the wrapper to `.bak` — any pre-install terminal silently lost its statusline because the stale per-handle override still won over the now-clean global, but pointed at a path that no longer existed.

**Fix:** extracted the per-file work into `strip_legacy_statusline_from_file(path) -> Result<bool>` and added `migrate_handle_dir_statuslines(base_dir) -> Result<Vec<u32>>` mirroring the existing `migrate_per_slot_statuslines` shape but iterating `term-<pid>` dirs. The install summary now reports both per-slot and per-handle migrations on separate lines:

```text
✓ Cleared stale per-slot statusLine on slot(s): 3, 7
✓ Cleared stale per-handle statusLine on terminal(s): 12345, 67890
```

User-custom commands continue to be preserved across both walks (only the known legacy wrappers `statusline-quota.sh` and `statusline-command.sh` are stripped).

## Migration & compatibility

**No schema changes.** `quota.json` stays at v2 (write-flip landed in v2.1.0). Credentials, handle dirs, and configuration files are compatible both ways between v2.1.0 and v2.1.1.

**Both migrations are silent and idempotent.** Re-running on already-clean state is a no-op (no mtime tick, no log noise above DEBUG).

**Out of scope for both migrations:** user-authored content. The apiKeyHelper migration touches only csq-managed files (`config-N/settings.json` + `settings-*.json` under `~/.claude/accounts/`); user-authored entries in `~/.claude/settings.json` or other paths are NOT touched. The statusline migration only strips known legacy wrapper commands; user-custom statusline commands are preserved.

## Upgrading

```
# Existing v2.1.0 users (tray)
Click "Update" in the csq tray menu.

# CLI users
curl -L https://github.com/terrene-foundation/csq/releases/download/v2.1.1/csq-<os>-<arch> -o /usr/local/bin/csq
chmod +x /usr/local/bin/csq
```

After upgrade:

- The apiKeyHelper migration runs automatically on the next daemon start. If you previously saw `apiKeyHelper failed: exited 127` warnings on CC launch, they should disappear.
- If you have any pre-existing terminals whose statusline disappeared after a recent `csq install`, run `csq install` once more — it will now strip the stale wrapper from those terminals' handle dirs and let the global statusline cascade through. CC reloads on mtime change so you'll see the statusline come back without restarting the terminal.
