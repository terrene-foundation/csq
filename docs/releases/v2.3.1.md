## What's new

v2.3.1 is a **patch release** on v2.3.0. It closes the desktop-side orphan-key risk that landed alongside the v2.3.0 Gemini surface, and rides one cosmetic csq-cli cleanup. No new features, no schema changes, no behavior change for users who never bind a Gemini account.

If you upgraded to v2.3.0 (Gemini surface) and intend to remove a Gemini account from the desktop UI, update.

### The fixes

#### D7 — desktop "remove Gemini account" now deletes the OS keychain entry

v2.3.0 shipped Gemini provisioning end-to-end. The CLI removal path (`csq logout N`) called `vault.delete(SlotKey { surface: SURFACE_GEMINI, account: N })` so the at-rest secret cleared with the slot. The desktop "remove account" flow (`remove_account` Tauri command) walked the marker file and `config-N/` directories but never touched the keychain — the `gemini/<slot>` entry outlived the slot, and a future bind to the same slot would silently inherit the orphan secret.

**Fix:** new `csq_core::providers::gemini::provisioning::delete_api_key_from_vault(base_dir, slot, vault) -> Result<bool, ProvisionError>`. Reads the binding marker to determine auth mode, calls `vault.delete` for `ApiKey` slots, no-ops for `VertexSa` slots (no vault entry exists), no-ops for absent markers (nothing to clean). Idempotent: a second call after the marker is gone returns `Ok(false)` without touching the vault.

`remove_account` (`csq-desktop/src-tauri/src/commands.rs`) now: (1) checks `is_gemini_bound_slot(base_dir, slot)`, (2) opens the vault and calls `delete_api_key_from_vault` BEFORE touching the marker, (3) calls `gemini_unbind` to remove the marker, (4) falls through to `logout_account` for any residual Anthropic-OAuth state. `LogoutError::NotConfigured` is treated as success when the Gemini marker was removed (covers the Gemini-only slot case where no `credentials/N.json` or `config-N/` exists).

Vault errors map via the existing `gemini_provision_api_key` error mapping — `BackendUnavailable` surfaces as "vault unavailable", never raw error bodies (`security.md` MUST 2). `delete_api_key_from_vault` is exercised by 4 csq-core unit tests (vault-clear happy path, VertexSa no-op, absent marker, idempotency) and 2 csq-desktop integration tests (Gemini slot removal clears the marker; non-Gemini slot never opens the vault).

The fix also adds `csq-core = { path = "...", features = ["secret-in-memory"] }` to `csq-desktop`'s `[dev-dependencies]`. `open_default_vault` gates the in-memory backend on `cfg!(any(test, feature = "secret-in-memory"))`, but `cfg!(test)` only fires when csq-core itself is being compiled with `--test`; loaded as a normal dep by csq-desktop's test build it is false, and Linux + Windows CI returned `BackendUnavailable` because no OS keyring is reachable in headless runners. Cargo unifies features across `[dependencies]` and `[dev-dependencies]` of the same crate, so the in-memory backend is now reachable in `cargo test` builds and stays absent from `cargo build --release`. macOS happened to pass under v2.3.0 because Keychain is reachable in the runner — the test was platform-flaky in v2.3.0; v2.3.1 makes it deterministic.

#### csq-cli — collapse Gemini orchestration to csq-core helpers

`csq-cli/src/commands/setkey.rs::provision_api_key` (lines 181-196) re-implemented `vault.set` + `GeminiBinding::new(ApiKey, "auto")` + `provisioning::write_binding` + on-failure `vault.delete` rollback. The csq-core helper `provision_api_key_via_vault(base_dir, slot, key, vault)` does this atomically. CLI now calls the helper; the `AIza` prefix guard, error mapping (`map_vault_error` / `map_provision_error`), and user-facing print output (fingerprint + launch hint) stay in csq-cli where they belong.

`csq-cli/src/commands/models.rs::write_gemini_model_to_binding` (line 273) re-implemented `read_binding` + field mutation + `write_binding`. The csq-core helper `set_model_name(base_dir, slot, model)` does this; CLI now delegates and maps `ProvisionError::Io { source }` with `kind() == NotFound` to the existing "no Gemini binding — run setkey first" error so the operator-facing message is unchanged.

No behavior change. ~30 LOC of duplication collapsed.

## Migration & compatibility

**No schema migration.** No on-disk artifact changes. v2.3.0 ↔ v2.3.1 are byte-identical for users who never invoke the desktop "remove Gemini account" flow.

**No behavior change for non-Gemini users.** The `remove_account` change short-circuits on the `is_gemini_bound_slot` check: ClaudeCode and Codex slots never reach the new vault-delete branch, so the OS keychain is never touched for those slots and the test environment carries no new credential dependency.

## Platform notes

Same as v2.3.0. No platform-specific code changes in this patch. The only platform-related change is the `secret-in-memory` dev-dep feature, which is test-build-only and does not affect release builds.

## Upgrading

```
# Existing v2.3.0 users (tray)
Click "Update" in the csq tray menu.

# CLI users
curl -L https://github.com/terrene-foundation/csq/releases/download/v2.3.1/csq-<os>-<arch> -o /usr/local/bin/csq
chmod +x /usr/local/bin/csq
```

After upgrade:

- If you have ever bound a Gemini slot and removed it from the desktop UI under v2.3.0, the OS keychain may carry an orphan `gemini/<slot>` entry. To clear it, re-bind that slot via `csq setkey gemini --slot <N> --from-stdin` and remove it again — v2.3.1's `remove_account` now cleans up correctly. Or run `csq logout <N>` from the CLI; the CLI path always cleared the vault.
