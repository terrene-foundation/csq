# Decision — PR-G4b: models switch gemini + cross-surface swap

## Context

PR-G4a (#198) shipped `csq setkey gemini` + `spawn_gemini` + `csq run`
dispatch + the H2 IPC slot guard. It explicitly deferred two
`/02-plans/01-implementation-plan.md` §PR-G4 deliverables:

- **`csq models switch --provider gemini --slot N <model>`** —
  atomic rewrite of the model field; existing branch in `models.rs`
  refused with "not yet implemented in this build" (PR-G1 stub).
- **Cross-surface `csq swap`** — `commands/swap.rs::cross_surface_exec`
  refused `Surface::Gemini` as a target with "swap to Gemini account
  is not supported in this build". Source-side detection had no
  `GEMINI_CLI_HOME` branch.

PR-G4b lands both, plus the four Gemini models and the synthetic
`auto` literal that FR-G-UI-02 / ADR-G08 mandate.

## Decisions

### D1 — `binding.model_name`, NOT `<config-N>/.gemini/settings.json`, is the persistent slot model state

The original FR-G-CLI-04 says "writes `model.name` in
`config-<N>/.gemini/settings.json` via atomic rename." But PR-G4a
chose a different layout: `<handle_dir>/.gemini/settings.json` is
regenerated by the EP1 drift detector on every spawn from
`binding.model_name`. There is no `config-<N>/.gemini/settings.json`
in the on-disk model.

So the persistent model state lives in the binding marker
`credentials/gemini-<N>.json`. `models switch` updates `model_name`
there; the next `csq run <slot>` picks up the new model via the
drift detector's `reassert_api_key_selected_type(_, model_name)`
call. The atomicity guarantee comes from `provisioning::write_binding`
(PR-G4a) which uses the `unique_tmp_path → secure_file →
atomic_replace` pipeline.

This is consistent with how Codex stores its model in
`config-<N>/config.toml` (different file, same per-slot persistent
location pattern) and how 3P providers store theirs in
`config-<N>/settings.json`.

### D2 — `auto` is a query-time literal, NOT a catalog entry

Four concrete Gemini models go in the catalog:

- `gemini-2.5-pro` (aliases: `pro`, `2.5-pro`)
- `gemini-2.5-flash` (aliases: `flash`, `2.5-flash`)
- `gemini-2.5-flash-lite` (aliases: `flash-lite`, `2.5-flash-lite`)
- `gemini-3-pro-preview` (aliases: `3-pro-preview`, `3-pro`)

`auto` is intentionally NOT a catalog entry because it is not a
model id — it instructs gemini-cli to pick. The dispatch path
short-circuits `auto` before the catalog lookup so the
suggestion-fallback (`did you mean claude-opus...`) does not surface
a misleading rejection. The binding marker stores the literal
`"auto"` and the drift detector writes that into the handle dir's
`.gemini/settings.json` `model.name` field, where gemini-cli treats
it as "pick automatically".

### D3 — Same-surface Gemini → Gemini takes the cross-surface exec path

ClaudeCode→ClaudeCode and Codex→Codex use in-flight symlink repoint
(both CLIs re-stat their credential file before each API call). But
gemini-cli does NOT re-read `GEMINI_API_KEY` mid-process — it
captures the env var at startup. So even a same-surface
Gemini→Gemini swap requires the new slot's vault key to reach a
fresh process.

The route table now sends Gemini→Gemini through `RouteKind::CrossSurface`
(tombstone source handle dir + exec gemini-cli with new vault key).
The `confirm_cross_surface` prompt is suppressed when source surface
== target surface, so Gemini→Gemini swaps execute silently like the
ClaudeCode→ClaudeCode case (just with different mechanics).

### D4 — `exec_gemini` is duplicated from `launch_gemini`, not factored

`commands/run.rs::launch_gemini` and `commands/swap.rs::exec_gemini`
both do the same ~20 LOC: verify binding marker is a regular file,
build minimal handle dir + `.csq-account` marker, open default
vault, call `spawn_gemini`. Factoring this into a `csq-core`
helper would require introducing a typed `SessionError` enum and a
new `gemini::session` module. Deferred until PR-G5 (desktop)
becomes the third caller — at which point the cost-benefit flips.

### D5 — `GEMINI_CLI_HOME` is the source-detection signal

`SourceHandle::Gemini(PathBuf)` joins the existing `ClaudeCode` and
`Codex` variants. `detect_source_handle` probes
`GEMINI_CLI_HOME → CODEX_HOME → CLAUDE_CONFIG_DIR` in that order;
the env var that points at a `term-<pid>` directory wins. Each
`csq run` path scrubs the other surfaces' env vars on launch, so
ordering only matters when a parent shell exports multiple of them
(unusual; the ordering picks the most-specifically-set variable).

### Spec touchpoints

The implementation deviates from FR-G-CLI-04's literal text ("writes
`model.name` in `config-<N>/.gemini/settings.json`"). The deviation
is documented in this journal — D1 above. The persistent state is
the binding marker; the handle-dir `.gemini/settings.json` is a
per-spawn artefact regenerated by the EP1 drift detector.

Spec 07 §7.2.3 (handle-dir layout) was already consistent with this
layout post PR-G4a; no edit needed.

### CLI surface

- `csq models switch gemini auto --slot N` → writes `auto`
- `csq models switch gemini pro --slot N` → writes `gemini-2.5-pro`
- `csq models switch gemini gemini-3-pro-preview --slot N` →
  writes id verbatim + stderr warning about preview tier
- `csq models list gemini` → lists 4 catalog entries
- `csq swap N` (from any surface to a Gemini-bound slot) → tombstone
  + exec gemini-cli
- `csq swap N` (from a Gemini slot to any other surface) → tombstone
  + exec target binary
- `csq swap N` (Gemini → Gemini) → tombstone + exec gemini-cli (no
  prompt; source surface == target surface)

## For Discussion

1. **D4 deferral threshold — when does `exec_gemini` factoring pay
   for itself?** Today there are two callers (run, swap). Factoring
   into `csq-core::providers::gemini::session::launch` requires a
   `SessionError` enum + module + 4 mapping branches in csq-cli that
   replace 2 inline anyhow! lines per caller. The break-even is
   ~3 callers OR a redteam finding flagging silent inconsistency
   between the two duplicates. Should the desktop UI in PR-G5 (the
   third potential caller) drive the factoring, or is there a case
   for doing it before PR-G5 to keep the surface stable for review?

2. **Counterfactual — if Gemini same-surface swap had been
   implemented as in-flight repoint anyway.** We could fork a
   "vault-poll" thread in spawn_gemini that re-reads the vault on
   SIGHUP and reissues the env var via... actually there's no Unix
   primitive for this — `GEMINI_API_KEY` is captured by gemini-cli
   at startup and there's no env-var-update API. So the "in-flight
   swap" alternative is not "more work" — it's "structurally
   impossible". This is interesting evidence that NOT every
   credential cache is in-flight-swappable; the pattern depends on
   the upstream CLI's re-stat / re-read discipline. CC re-stats
   `.credentials.json` per spec 01 §1.4; Codex re-stats `auth.json`
   per spec 07 §7.2.2; gemini-cli does neither.

3. **Evidence question — does the catalog provider check correctly
   refuse Claude → Gemini cross-provider model switches?**
   `handle_switch` calls `catalog.find(model_query)` then checks
   `m.provider != provider_id` for non-ollama / non-codex paths.
   Adding Gemini to the catalog means a user running `csq models
   switch claude pro` would resolve `pro` → `gemini-2.5-pro` (alias
   match) and then refuse with "model gemini-2.5-pro belongs to
   provider gemini, not claude". The `pro` alias is unique to
   Gemini in the catalog after PR-G4b. But "claude" provider's
   aliases include `opus`, `sonnet`, `haiku` — none of which collide
   with Gemini. So the cross-provider refusal works. Worth a
   regression test in PR-G5 if that test surface is touched again.
