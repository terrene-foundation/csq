# 0049 — DISCOVERY — CI cache poisoned by the `claude-squad → csq` rename

## Symptom

From PR #88 onward, **every `push main` and `pull_request` run of `.github/workflows/test.yml`** failed with:

```
failed to read plugin permissions: failed to read file
'/Users/runner/work/claude-squad/claude-squad/target/debug/build/tauri-…/out/permissions/app/autogenerated/commands/app_hide.toml':
No such file or directory (os error 2)
```

On Windows the same error, with path `D:\a\claude-squad\claude-squad\...`.

The _release_ workflow (`release.yml`, triggered on tag push) kept passing, so tagged releases like v2.0.0-alpha.13 shipped green while every PR after #88 landed on red main. The Foundation merged through it with `--admin`.

## Root cause

`actions/cache@v4` key in `test.yml`:

```yaml
key: ${{ runner.os }}-cargo-${{ env.CACHE_VERSION }}-${{ env.RUST_VERSION }}-${{ hashFiles('**/Cargo.lock') }}
```

`CACHE_VERSION` was pinned to `v1` and `Cargo.lock` didn't change for the rename, so restores kept returning the old cache.

GitHub Actions workspace path is `/home/runner/work/<REPO>/<REPO>/`. When the repo was `claude-squad` the workspace was `/home/runner/work/claude-squad/claude-squad/`. Tauri's build script writes plugin permission files at absolute paths **into `target/debug/build/tauri-*/out/…`**, and those paths are compiled into the cached build graph.

After the rename, cache restores put the old workspace-path strings back into the new `/home/runner/work/csq/csq/target/…` tree. Every subsequent build tried to read files at `…/claude-squad/claude-squad/…` — a path that no longer exists.

## Fix

One-line bump in `.github/workflows/test.yml`:

```yaml
# v2: busted after the repo rename from `claude-squad` → `csq`.
CACHE_VERSION: v2
```

Landed in PR #113 (this session). Post-bump, `target/` rebuilds from scratch with correct paths and every platform builds cleanly.

## Why this surfaced now (2026-04-14, not earlier)

Developers and CI had been merging over red CI for months because:

1. Local `cargo test` + `cargo clippy` worked (different cache scope, no repo-path indirection).
2. Release tagging used `release.yml` which keys its cache separately.
3. Nobody ran `gh run list --workflow=test.yml` to notice the streak.

I noticed only because I asked "did my alpha.14 PRs pass CI?" and the history showed every post-#88 run as `failure`.

## Downstream discoveries

Busting the cache made the build succeed for the first time in months, which surfaced **five latent test failures** that had been hidden by the earlier build stage failing first. Those are documented in journal 0050.

## How to apply

- On any future repo rename, **bump `CACHE_VERSION` in every workflow that caches `target/`** in the same PR as the rename. Treat the cache-version bump as part of the rename checklist.
- A cargo cache key that includes `runner.workspace` or `github.repository` would prevent this class of poisoning entirely, but the bump is fine as a tactical fix.
- Monitor `gh run list --workflow=test.yml --limit 20` periodically — not just at release time. An always-red main CI rots quickly because nobody trusts it.
