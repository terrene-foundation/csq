# csq v2.3.0

**Release date:** 2026-04-26
**Prior release:** v2.2.0 (2026-04-25)
**Checksum / signature:** see artifact `SHA256SUMS` and `.sig` files on the GitHub Releases page.

## What's new

v2.3.0 adds **Gemini** as a first-class third surface alongside ClaudeCode and Codex. csq now manages Google Gemini accounts end-to-end: API-key provisioning (AI Studio paste or Vertex SA JSON path) under encryption-at-rest, in-flight `csq swap` between Gemini slots, cross-surface swap between Gemini and the other two surfaces, `csq run` that does **not** require a running daemon (a deliberate inversion of v2.1.0's INV-P02 for Codex), live event-driven quota with a CLI-durable NDJSON event log, and a desktop UI that mirrors the ClaudeCode and Codex flows with a Terms-of-Service disclosure for the first-time Gemini sign-in.

The release also lands a new `platform::secret` encryption-at-rest primitive (macOS Keychain, Linux Secret Service with AES-GCM file fallback, Windows DPAPI / Credential Manager), a 7-layer Terms-of-Service defense (EP1–EP7) pinned to a specific `gemini-cli` minor release, and a CLI-durable NDJSON event log that decouples Gemini quota updates from daemon availability.

If you do not use Gemini, you can update with no behaviour change on the ClaudeCode or Codex side. There is no schema migration: `quota.json` stays at v2, credential files, handle dirs, `settings.json` files, and daemon socket protocol are compatible both directions between v2.2.0 and v2.3.0.

### The Gemini surface

- **API-key only** (ADR-G09). Gemini ships without OAuth subscription rerouting per Google's Terms of Service. Provisioning is via AI Studio key paste or Vertex SA JSON path; both flow through `platform::secret` and never reach process argv. A 7-layer defense (EP1–EP7) actively prevents the OAuth-rerouting failure mode at runtime — no disable knob, no "advisory" mode (C-CR1).
- **`Surface::Gemini` enum** (PR-G1, PR-G2b). The surface dispatch architecture from v2.1.0 extends to a third variant. `discovery`, `auto_rotate`, `rotation::swap_to`, `daemon::refresher`, and `usage_poller` route Gemini slots through their Gemini-specific paths. PR-G2b flips every `platform::secret` literal to use `Surface::Gemini.as_str()` so the secret backend's namespace is owned by the surface enum.
- **`platform::secret` primitive** (PR-G2a, PR-G2a.2, PR-G2a.3). New encryption-at-rest module with five backends: `macos.rs` (Keychain via `security-framework`), `linux.rs` (Secret Service via D-Bus, with AES-GCM file fallback when no keyring is available), `windows.rs` (DPAPI + Credential Manager), `file.rs` (AES-GCM-only fallback for headless / CI / WSL-no-keyring environments), and `in_memory.rs` (test-only). Every backend implements the same `SecretStore` trait; `audit.rs` carries the security-reviewer sign-off ledger required by spec 07 §7.2.3.
- **Gemini login** (PR-G4a). `csq setkey gemini N --from-stdin` on the CLI and the desktop AddAccountModal both run a stdin-only provisioning flow. The API key never appears in argv, never lands in any structured log, and is redacted by `error::redact_tokens` — the redactor learns the `AIza*` prefix in this release. Canonical secret lands in `platform::secret` at namespace `gemini/<slot>`; the handle dir's `.gemini/oauth_creds.json` and any pre-existing OAuth residue are detected and refused (EP1, EP2).
- **No daemon required for `csq run`** (ADR-G09, INV-P02 inverted). Unlike Codex slots, Gemini `csq run N` works without a running csq daemon. Quota is event-driven via the NDJSON event log (below), so the daemon is only needed for centralized aggregation, not for spawning a session. A fresh-install user can `setkey` + `run` a Gemini slot without any daemon supervision running.
- **NDJSON event log** (PR-G3). Gemini quota is event-driven, not poll-driven. The CLI writes one `accounts/gemini-events-<slot>.ndjson` per slot per session, opened with `O_APPEND` and `fsync` after every record (durability floor). The daemon drains these on startup and on every reconnect, advances `quota.json` accordingly, and prunes drained records. The single-writer-to-quota.json invariant is preserved: only the daemon writes quota. Corrupted event logs rotate to `gemini-events-<slot>.corrupt.<unix_ms>` and the slot starts fresh.
- **Cross-surface `csq swap`** (PR-G4b). `csq swap N` resolves the target's surface from `discovery::discover_all` and routes Gemini-bound swaps to a Gemini-aware path. Same-surface Gemini→Gemini swaps repoint atomically; cross-surface (e.g., ClaudeCode → Gemini) follow the v2.1.0 INV-P05 confirm prompt + INV-P10 rename-source-to-tombstone + `exec` of the target binary path.
- **`csq models switch` Gemini dispatch** (PR-G4b). `csq models switch <slot> <model>` classifies Gemini slots via `discover_all` and routes them to a static catalog with four entries (`gemini-2.5-pro`, `gemini-2.5-flash`, `gemini-2.0-flash-exp`, `gemini-1.5-pro`). The catalog is static because gemini-cli does not expose a `/models` endpoint analogous to `claude-cli`'s; new entries land in csq-core releases.
- **Desktop Gemini UI** (PR-G5). AddAccountModal gains a Gemini tab with two sub-tabs (AI Studio key paste / Vertex SA JSON file picker) plus a ToS disclosure modal and an inline OAuth-residue warning panel (FR-G-UI-01). ChangeModelModal renders the static Gemini model list with a preview-tier downgrade warning (FR-G-UI-02). AccountCard renders a Gemini surface badge, a downgrade chip when the user is on a preview tier, and "quota: n/a" until the first NDJSON event drains (FR-G-UI-03). Six new Tauri commands: `is_gemini_tos_acknowledged`, `acknowledge_gemini_tos`, `gemini_probe_tos_residue`, `gemini_provision_api_key`, `gemini_provision_vertex_sa`, `gemini_switch_model`.

### 7-layer Terms-of-Service defense (EP1–EP7)

C-CR1 mandates active enforcement: no disable knob, no "EP4 is advisory" reclassification. The seven layers are independent and all mandatory.

- **EP1 — pre-spawn provisioning gate.** `csq setkey gemini` refuses to provision when `~/.gemini/oauth_creds.json` is present, surfacing the path and a remediation hint. Same gate runs in `gemini_provision` Tauri command before reaching the secret backend.
- **EP2 — pre-spawn handle-dir audit.** `spawn_gemini` calls `reassert_api_key_selected_type(handle_dir)` immediately before `exec`, ensuring `~/.gemini/settings.json` carries `selectedType=api-key` and refusing to spawn if user-level `oauth-personal` would override (OPEN-G01 outcome, journal 0003).
- **EP3 — environment scrub.** `Command::env_remove` drops `GOOGLE_APPLICATION_CREDENTIALS`, `GEMINI_API_KEY`, and the rest of the OAuth-precedence env set before `exec` (journal 0004 captured the precedence order live).
- **EP4 — runtime stderr sentinel.** `tos_guard.rs` scans gemini-cli's stderr for the documented OAuth-rerouting markers (`google-gemini/gemini-cli` issues #20632 and #22970 confirmed active 403 enforcement). Markers are pinned to **gemini-cli 0.38.x** verbatim; a minor-version bump triggers either an auto-update of the whitelist or a version-mismatch dialog (PR-G3).
- **EP5 — silent-downgrade detector.** Response-shape capture (live, journal 0004) flags responses whose `modelVersion` does not match the slot's selected model. Detector runs against REST `generateContent` and SSE `streamGenerateContent`. Debounce is on the receive side per spec 05 §5.8 (csq-core `providers::gemini::capture`).
- **EP6 — `.env` short-circuit scan.** Pre-spawn scan of `$CWD/.env` for `GEMINI_API_KEY=` / `GOOGLE_API_KEY=` lines that would leak the key into the spawned process env. Refusal is hard, not advisory.
- **EP7 — vault-delete on unbind.** `csq logout N` + the desktop "remove Gemini account" path call `platform::secret::vault.delete(gemini/<slot>)` so an orphaned secret never outlives its slot.

The whitelist regression test is green; an EP4 verbatim-marker change in any future gemini-cli minor release will fail the test before any user can bypass the sentinel. Versioned whitelist is anchored at `0.38.x` in this release.

### NDJSON event log — CLI-durable quota path

The Gemini event-delivery contract is frozen in **spec 07 §7.2.3.1** (PR-G0). Key rules:

- Socket-path resolution: `$XDG_RUNTIME_DIR/csq.sock` or `~/.claude/accounts/csq.sock`, shared with spec 04 §4.2.5 layer 3 via `platform::paths::daemon_socket()`.
- Non-blocking connect ceiling: 50 ms. Above this, the emitter drops to NDJSON-as-durability-floor (write the record to disk, do not block the spawn).
- Drop-on-unavailable semantics: emitter MUST NOT block on socket. If the daemon is down, the record lands on disk only; the next daemon tick drains it.
- File format: one JSON object per line, ASCII-only, terminated by `\n`. Slot encoded in filename, not body. Records carry timestamp + event-kind + opaque payload — no API keys, no PII.
- File permissions: `0600`. Directory: `accounts/`. Files are gitignored.

The drain path is exercised by an integration test that takes the daemon down, runs a CLI session that emits 5 events, brings the daemon back up, and asserts `quota.json` reflects the queued events. Durability is verified end-to-end.

### Migration & compatibility

**No schema migration.** `quota.json` stays at v2. Credential files, handle dirs, `settings.json` files, and daemon socket protocol are compatible both directions between v2.2.0 and v2.3.0.

**Gemini Terms-of-Service disclosure.** First-time Gemini sign-in via the desktop UI presents a Terms-of-Service screen that the user must acknowledge before the API key can be provisioned. The marker is persisted at `accounts/.gemini-tos-acknowledged-<slot>` and re-checked on every `gemini_provision` invocation. Acknowledgement is per-slot, not global.

**No daemon requirement for Gemini.** Unlike Codex, Gemini `csq run N` works without a running csq daemon. Quota updates are event-driven via the NDJSON log and the daemon drains them on its next tick. Centralized refresh / auto-rotate are not applicable to Gemini (API keys do not expire).

**New configuration files.**

- New file: `accounts/gemini-events-<N>.ndjson` (per-slot event log, 0o600, gitignored, drained by daemon).
- New file: `accounts/gemini-events-<N>.corrupt.<unix_ms>` (rotated event log on parse failure, retained for forensic review).
- New file: `accounts/.gemini-tos-acknowledged-<N>` (per-slot ToS acknowledgement marker).
- New keychain namespace: `gemini/<slot>` under `platform::secret` (Keychain on macOS, Secret Service on Linux, Credential Manager on Windows, AES-GCM file fallback when no OS keyring is available).

### Deferred

- **D7 — vault-delete-on-unbind from desktop** (journal 0011 §FD #1, restated in 0013). The CLI `csq logout` path calls `vault.delete` on Gemini slot removal. The desktop "remove Gemini account" flow does not yet — this lands as a follow-up so an orphaned `gemini/<slot>` secret cannot outlive the slot in the desktop-only workflow.
- **csq-cli orchestration cleanup** (journal 0013 §"What's left"). `setkey.rs::handle_gemini` and `models.rs::write_gemini_model_to_binding` carry a small parallel implementation of helpers that csq-core also exposes (`provision_api_key_via_vault`, `set_model_name`). ~30 LOC of contained duplication; collapses to single-source after the post-release follow-up.
- **`launch_gemini` / `exec_gemini` factoring** (journal 0012 §D4). Today there are 2 callers (`commands/run.rs` + `commands/swap.rs`). The factoring threshold is 3; deferred until v2.4 brings an in-app launch button OR Bedrock surface.

## Platform notes

**macOS.** Same as v2.2.0 — `.app` bundle is ad-hoc signed (no Apple Developer cert provisioned). Right-click → Open required on first launch on Sequoia / Sonoma, or `xattr -d com.apple.quarantine "/Applications/Code Session Quota.app"` to drop the Gatekeeper attribute. `.dmg` signing is consistent (codesign --force --deep --sign - + hdiutil create) so the app is launchable without a re-sign step after install. `platform::secret` macOS backend uses `security-framework` and lands every secret in the user's login Keychain at service `foundation.terrene.csq.gemini`.

**Linux.** `platform::secret` Linux backend tries Secret Service first (D-Bus to `org.freedesktop.secrets`); on systems with no keyring (typical headless / CI / minimal WSL), it falls back to AES-GCM-encrypted file at `~/.local/share/csq/secret/<surface>/<slot>.aes-gcm` with a key derived from the user's machine ID. Daemon socket at `$XDG_RUNTIME_DIR/csq/csq.sock` (or `~/.claude/accounts/csq.sock` if XDG_RUNTIME_DIR is unset); `SO_PEERCRED` enforces same-UID. Gemini surface tested on x86_64 Linux + WSL2.

**Windows.** Daemon supervisor still short-circuits per v2.0.1 PR-VP-C1a (the windows-daemon Cargo feature is default-off). Therefore:

- Gemini on Windows is **supported** for `csq run` and `csq setkey` because Gemini does not require a daemon. `platform::secret::windows.rs` uses DPAPI for at-rest encryption + Credential Manager for namespaced storage.
- Codex on Windows remains **not supported** in v2.3 (Codex requires the daemon, which is short-circuited).
- ClaudeCode on Windows works as in v2.2.0 (no daemon-dependent features). `csq swap` on ClaudeCode handle dirs works in-flight.
- Cross-surface `csq swap` on Windows is untested and will fail at the `exec` step (`std::os::unix::process::CommandExt` is Unix-only). Same-surface Gemini→Gemini repoint on Windows is also Windows-untested per the v2.1.0 L-CDX-2 carry-over.

The full Windows port lands when PR-VP-C1b flips the daemon flag + the named-pipe Codex transport surface lands.

## Upgrading

```
# Existing v2.x.x users (tray)
Click "Update" in the csq tray menu.

# CLI users
curl -L https://github.com/terrene-foundation/csq/releases/download/v2.3.0/csq-<os>-<arch> -o /usr/local/bin/csq
chmod +x /usr/local/bin/csq
```

**No configuration migration required for ClaudeCode-only or Codex-only setups.** Quota.json stays at v2. Handle dirs and credentials are untouched.

**Gemini sign-in (new in v2.3).** After upgrade, open the desktop AddAccountModal and select the "Gemini" provider. The first-time ToS panel will appear; acknowledge to proceed to the API-key paste / Vertex SA picker. CLI users can run `csq setkey gemini <slot> --from-stdin` (key piped on stdin; never on argv).

If you have a pre-existing `~/.gemini/oauth_creds.json` from a prior gemini-cli OAuth login, EP1 will refuse provisioning and surface the path. The remediation is documented in the README Troubleshooting section.
